# Web3 學習筆記｜第 15 章：安全——開發者必修

> 學習目標：寫合約時會本能地問「這裡有什麼可以被利用」
> 完成日期：2026/06

---

## 核心問題

智能合約部署上鏈就改不掉，一旦有漏洞，損失可能是幾億美元且無法挽回。怎麼在部署前就把問題找出來？

> 一般軟體有 bug 可以推送更新修掉。智能合約不行——部署上鏈之後程式碼永久存在，你只有一次機會把它做對。

---

## 核心概念一：重入攻擊（Reentrancy Attack）

### 問題根源：步驟順序錯了

錯誤的提款邏輯：

```
步驟一：確認餘額夠不夠
步驟二：把錢轉出去      ← 先給錢
步驟三：把餘額歸零      ← 才扣款（太晚了）
```

關鍵漏洞：當合約把 ETH 轉給另一個合約地址時，接收方合約可以在「收到 ETH 的瞬間」自動執行程式碼，再次呼叫提款——而此時餘額還沒歸零。

### 攻擊流程

```
攻擊者呼叫提款
→ 受害合約確認餘額（1 ETH）→ 轉出 1 ETH
→ 攻擊者合約收到 ETH → 自動再呼叫提款
→ 受害合約確認餘額（還是 1 ETH，還沒歸零）→ 再轉出 1 ETH
→ 一直重複，直到受害合約的錢被掏空
```

### 修復方式：CEI 原則（Checks-Effects-Interactions）

```
正確順序：
步驟一：確認（Check）餘額夠不夠
步驟二：更新狀態（Effect）把餘額歸零   ← 先做這個
步驟三：互動（Interaction）把錢轉出去
```

先扣款再給錢——就算攻擊者再呼叫一次，餘額已經是零，直接被拒絕。

### 歷史案例

2016 年 The DAO 事件，損失約 $6,000 萬美元的 ETH，最終導致以太坊分叉成兩條鏈（ETH 和 ETC）。

## 核心概念二：權限失控（Access Control Vulnerability）

### 常見的三種情況

```
情況一：忘記設限制
→ 任何人都能呼叫關鍵函數（例如提款、鑄幣）

情況二：限制設錯了
→ 邏輯反了，幾乎所有人都能操作

情況三：有後門
→ 開發者留了「管理員功能」可以隨時提走合約裡的錢
→ 用戶不知道這個後門存在
```

### 修復方式

```solidity
// 危險：任何人都能呼叫
function withdrawAll() public {
    transfer(msg.sender, address(this).balance);
}

// 安全：只有 owner 能呼叫
function withdrawAll() public onlyOwner {
    transfer(msg.sender, address(this).balance);
}
```

特別注意**初始化函數（initialize()）**——如果沒有限制「只能呼叫一次」，任何人都能把自己設定成合約擁有者。

### PM 的判斷點

```
問工程師：
→ 關鍵函數有沒有權限控制？
→ 升級權限在誰手上？
→ 有沒有緊急暫停功能？誰能觸發？

注意：如果升級或暫停權限集中在開發團隊單一地址
→ 這是中心化風險，跟「去中心化」的承諾矛盾
```

## 核心概念三：Front-running（搶先交易）

### 根本原因

交易在上鏈之前會先進 Mempool（公開等待區），任何人都能看到還沒上鏈的交易內容。

### 三明治攻擊（Sandwich Attack）

以你在 Uniswap 用 10 ETH 換 USDC 為例：

```
正常情況：10 ETH 換到 30,000 USDC

被三明治攻擊後：

第一步：機器人看到你的交易，搶先買入 USDC
→ 池子裡 USDC 變少，價格被推高

第二步：你的交易上鏈
→ 你用更高的價格換，只換到 29,500 USDC

第三步：機器人立刻賣出 USDC 獲利
→ 賺走 500 USDC 的價差
```

你被夾在機器人的兩筆交易中間——這就是「三明治」。

### 機器人具備什麼？

```
① 能看到 Mempool（公開資訊，架設節點即可）
② 能快速計算這筆攻擊是否有利潤（自動化程式）
③ 能付更高的 Gas 費搶先上鏈
```

機器人沒有做任何「違法」的事——它只是比你快地利用了公開資訊。

### 這個行為的學術名稱：MEV

**MEV（Maximal Extractable Value，最大可提取價值）**：驗證者或機器人透過控制交易順序來獲利。

### 防禦方式

```
滑點保護（Slippage）：設定最多接受價格偏差 X%
→ 被 front-run 導致偏差超標 → 交易自動失敗

Private Mempool：交易不經過公開 Mempool
→ 機器人看不到

Commit-Reveal：先提交加密的意圖，確認後再揭露
→ 機器人無法預先看到你要做什麼
```

Front-running 很難被完全消除，只能被緩解——因為它利用的是區塊鏈公開透明的基本特性。

## 核心概念四：預言機操縱（Oracle Manipulation）

### 閃電貸攻擊（Flash Loan Attack）

閃電貸規則：同一筆交易裡可以借出大量資金，但交易結束前必須還回去。

攻擊者把閃電貸和預言機操縱結合：

```
同一筆交易裡：
步驟一：閃電貸借出 1 億美元（不需要自己有錢）
步驟二：用 1 億美元在小型 DEX 上砸低某代幣價格
步驟三：預言機讀到這個假價格
步驟四：利用假價格在借貸協議裡做套利
步驟五：還回閃電貸
→ 整個攻擊在一個區塊內完成，淨賺價差
```

### 防禦方式

```
用多個預言機來源取平均值（不只接一個 DEX）
使用 TWAP（時間加權平均價格）
→ 即使瞬間被砸低，平均值不會大幅偏移
```

## 核心概念五：安全審計（Security Audit）

### 為什麼需要第三方審計

開發者寫程式碼時有天然盲點——你不自覺地按照「預設的使用方式」測試，看不到攻擊者用「奇怪的方式」使用你的合約。**球員兼裁判，永遠有盲點。**

### 審計流程

```
第三方審計公司拿到合約原始碼
→ 用攻擊者思維找漏洞
→ 出具報告：列出問題和嚴重程度
→ 開發團隊修復
→ 重新審計確認修復有效
```

知名審計公司：Trail of Bits、OpenZeppelin、Certik

### 審計不是萬能的

```
只能找到「已知類型」的漏洞
→ 全新攻擊手法審計師也可能沒見過

審計是某個時間點的快照
→ 合約升級後，新增程式碼沒被審計

審計品質差異很大
→ Certik 曾被批評品質不一致
```

### PM 的評估框架

```
有審計報告 → 基本門檻，還要看：
→ 哪家審計公司？（Trail of Bits、OpenZeppelin 是頂尖）
→ 高危漏洞都修復了嗎？
→ 審計後合約有沒有被修改？

沒有審計報告 → 警訊，尤其是管理大量資金的合約
```

## ✓ 學完能做到：評估 Agent Payment 合約安全性

看到以下描述時的疑慮：

> 「提款函數：先檢查餘額、把 ETH 轉給 agent、再把餘額歸零。接一個 DEX 的價格預言機計算匯率。通過 Certik 審計。」

```
疑慮一：提款順序錯了
→ 先轉錢再歸零 = 重入攻擊風險
→ 正確應該是先歸零再轉錢（CEI 原則）

疑慮二：只接單一 DEX 的預言機
→ 流動性低的 DEX 容易被閃電貸操縱
→ 為什麼不用 Chainlink 多來源聚合？

疑慮三：Certik 審計品質不穩定
→ 需要看報告細節
→ 確認高危漏洞都修復了
→ 確認審計後合約沒有被修改
```

---

## 與 Agent Payment 的連結

- **重入攻擊：** Agent payment 合約管理用戶資金，提款邏輯必須遵守 CEI 原則，否則 agent 的每次付款都可能是攻擊入口
- **權限失控：** Agent 的簽名權限設計必須有嚴格的存取控制，防止任何人偽造 agent 的付款指令
- **Front-running：** Agent 自動執行的交易模式可預測，機器人更容易針對 agent 的交易做三明治攻擊——agent payment 方案需要內建滑點保護
- **預言機操縱：** Agent 需要即時匯率換幣付款，單一來源的預言機是高風險——應使用多來源聚合預言機
- **安全審計：** 評估 agent payment 方案時，合約安全性是核心指標，要看審計公司品質和漏洞修復狀況
