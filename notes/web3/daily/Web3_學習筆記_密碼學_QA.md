## Web3 帳戶系統框架

- **Web2**：帳號密碼存在平台資料庫，平台說你有資產你才有
- **Web3**：用數學証明你擁有這個錢包，不需要任何平台同意
- 好處：開放、可組合、無需平台許可
- 代價：用戶自己承擔私鑰與簽名風險

---

## 一、私鑰（Private Key）＝ 控制權

- 一串隨機產生的數字，是 Web3 世界的「終極密碼」
- 誰有私鑰，誰就完全控制這個錢包
- 丟失私鑰 → 失去控制權（無法找回）
- 外洩私鑰 → 別人獲得控制權（無法阻止）

**運營視角**：CEX 的核心價值就是幫用戶保管私鑰，換取便利性；
信任感 ＝ 留存率的核心

---

## 二、公鑰（Public Key）與地址（Address）

### 單向推導鏈（不可逆）

```
私鑰（Private Key）
    ↓ 橢圓曲線數學運算（ECDSA）
公鑰（Public Key）
    ↓ 雜湊 + 取後幾碼
地址（Address）  例：0xAbCd...1234
```

| 方向 | 可以嗎？ |
|------|---------|
| 私鑰 → 公鑰 | ✅ 可以 |
| 公鑰 → 私鑰 | ❌ 數學上不可能 |
| 公鑰 → 地址 | ✅ 可以 |
| 地址 → 公鑰 | ❌ 不可能 |

→ 可以公開地址給別人匯款，對方無法反推私鑰

---

## 三、數位簽章（Digital Signature）＝ 授權證據

### 簽名流程
```
私鑰 + 交易內容 → 簽名演算法 → 簽章（Signature）
```

### 驗証流程（任何人都能做）
```
公鑰 + 簽章 + 交易內容 → 驗証演算法 → ✅ 有效 / ❌ 無效
```

- 私鑰從不公開，但任何人都能用公鑰驗証簽名
- 沒有私鑰，任何人都無法產生「通過公鑰驗証」的簽章
- **數學簽名 ＝ 數位簽章**，強調靠數學保證，不靠第三方

⚠️ 風險：盲簽（Blind Signing）是常見釣魚手法，
簽名內容必須讓用戶看得懂，運營設計活動時 UI 文案責任重大

---

## 四、雜湊（Hash）＝ 承諾

- 任何資料 → 雜湊函數（如 SHA-256）→ 固定長度「指紋」
- 原始資料改一個字，指紋就完全不同
- 只看指紋，無法反推原始資料

### 雜湊驗証原理
```
發送方：原始資料 + 雜湊值
接收方：對原始資料自己跑一次相同雜湊函數
比對結果：
  ✅ 一樣 → 資料完整，沒被竄改
  ❌ 不一樣 → 資料被動過
```

> 雜湊函數是公開標準，秘密不在演算法，在原始資料本身

### 區塊鏈串鏈原理

```
區塊 #1           區塊 #2              區塊 #3
┌───────────┐    ┌──────────────┐    ┌──────────────┐
│ 交易資料   │    │ 交易資料      │    │ 交易資料      │
│ 前一塊雜湊 │◀───│ 前一塊雜湊    │◀───│ 前一塊雜湊    │
│ （000）    │    │ （#1的值）   │    │ （#2的值）   │
└───────────┘    └──────────────┘    └──────────────┘
```

- 每個區塊存著上一個區塊的雜湊值
- 竄改任何一塊 → 後續所有區塊全部失效
- 這就是「帳本不可竄改」的數學基礎

| 概念 | 一句話 | 丟失/出錯的代價 |
|------|--------|----------------|
| 私鑰 | 控制權的鑰匙 | 丟了沒救、外洩被盜 |
| 公鑰 | 由私鑰推導，可公開 | 無法反推私鑰，安全 |
| 地址 | 由公鑰推導，錢包門牌 | 公開給人匯款即可 |

## 五、助記詞（Mnemonic）＝ 私鑰的備份

- 12 或 24 個英文單字，是人類可讀的私鑰備份形式
- **不是「猜測」私鑰，而是「推導」私鑰**——確定性的數學運算

```
助記詞（12/24 個單字）
    ↓ BIP-39 演算法
種子（Seed）
    ↓ BIP-32 演算法
主私鑰
    ↓ 依路徑衍生
帳戶 #1 私鑰 → 公鑰 → 地址
帳戶 #2 私鑰 → 公鑰 → 地址
……
```

- 同樣的助記詞，永遠推導出同樣的私鑰
- 助記詞外洩 ＝ 所有帳戶的私鑰全部外洩（比單一私鑰更危險）
- 紙抄下來就能備份整個錢包

## 六、地址驗証機制（ecrecover）

節點怎麼確認「這筆交易是地址持有者發出的」：

```
簽章 + 交易內容
    ↓ ecrecover（數學還原）
推算出公鑰
    ↓ 雜湊 + 取後幾碼
算出地址

比對：
算出的地址 == from address？
✅ 一致 → 確認是本人授權
❌ 不一致 → 交易無效
```

- 驗的不是地址本身，而是「簽章跟地址是否對得起來」
- 不需要任何人工審核，純數學自動完成

## 七、一筆交易的完整流程

### 階段一：錢包端（發起交易）
```
用戶輸入：收款地址 + 金額
錢包補上：nonce、gas price、gas limit、chain ID
    ↓
打包成交易 → 做雜湊 → 用私鑰簽名 → 產生簽章
    ↓
廣播出去：{ 交易內容 + 簽章 }
```

### 階段二：Mempool（等候區）
節點收到後各自驗証：
1. **簽章驗証**：ecrecover 算出地址 == from address？
2. **餘額檢查**：付得起金額 + 手續費？
3. **nonce 檢查**：序號是否正確？

全部通過 → 進入 Mempool，並轉發給其他節點

### 階段三：打包進區塊
```
驗証者從 Mempool 挑交易（優先高 gas price）
→ 逐一執行：扣款、加款、收手續費
→ 打包成新區塊（含前一塊雜湊值）
→ 廣播給全網
```

### 階段四：全網驗証
```
其他節點各自驗証：
① 前一塊雜湊值正確？
② 每筆交易合法？
③ 執行結果與狀態根雜湊一致？
全部通過 → 接受，加到自己的鏈上
```

### 階段五：確認完成
- 後續區塊疊越多，篡改成本越高
- 交易所通常要求 6～20 個確認才入帳

## 八、Nonce ＝ 交易序號

- 每個地址從 0 開始計數，每發一筆交易 +1
- 節點只接受「下一個該用的號碼」

| 狀況 | 結果 |
|------|------|
| nonce 已用過（太小） | ❌ 直接拒絕 |
| nonce 剛好是下一個 | ✅ 接受執行 |
| nonce 跳號（太大） | ⏳ 進 Mempool 等待 |

**為什麼需要 nonce：**
- 防止重放攻擊：舊交易廣播第二次，nonce 對不上 → 自動拒絕
- 保證交易順序：多筆交易一定按序號執行

---

## 九、多簽（Multi-signature / Multisig）

**一筆交易需要多把私鑰同時簽名才能執行。**

常見設定 2-of-3：有 3 把鑰匙，任意 2 把才能動錢

```
私鑰 A 被盜 → 攻擊者只有 1 把，動不了錢 ✅
私鑰 B 遺失 → A + C 還在，仍能正常使用 ✅
正常轉帳   → A + B 一起簽名，通過 ✅
```

**運營應用：**
- CEX 冷錢包幾乎都用多簽（財務長、技術長、CEO 各持一把）
- DAO 金庫用多簽，防止單一成員捲款
- 對外展示多簽設定 → 建立用戶信任感

## 十、帳戶抽象（Account Abstraction / AA）

讓錢包「變聰明」，用程式碼定義規則，不再只靠一把私鑰控制一切。

**原本以太坊兩種帳戶：**

| | 一般帳戶（EOA） | 合約帳戶 |
|-|---------------|---------|
| 控制方式 | 私鑰 | 程式碼 |
| 能主動發交易 | ✅ | ❌ |
| 自訂邏輯 | ❌ | ✅ |

**帳戶抽象（ERC-4337）能做到：**

| 功能 | 說明 |
|------|------|
| 社交恢復 | 私鑰丟了，讓朋友投票幫你恢復 |
| 別人幫付手續費 | 平台幫新用戶代付 gas |
| 批次交易 | 一次簽名執行多筆操作 |
| 消費限額 | 設定每天最多動多少錢 |
| Session Key | 暫時授權鑰匙，不必每次簽名 |

**運營應用：**
- 別人代付 gas → 降低新用戶入門門檻 → 提升拉新轉化
- 批次交易 → 「一鍵領取所有獎勵」體驗更順暢
- 社交恢復 → 解除「怕丟私鑰不敢用」的心理障礙 → 提升留存

## 十一、權限邊界（Permission Boundary）

**預先設定「這個 Agent 最多能做什麼」，超出範圍的操作它碰不了。**

針對 AI Agent 自動簽名的風險：AI 可能判斷錯誤、被 Prompt Injection 操控，若有無限授權，一個錯誤就能清空錢包。

**常見設定：**

| 邊界類型 | 說明 |
|---------|------|
| 金額上限 | 單筆最多 100 USDT |
| 時間限制 | Session Key 24 小時後自動失效 |
| 合約白名單 | 只能跟指定合約互動 |
| 操作類型限制 | 只能讀取，不能發出轉帳 |
| 人工確認門檻 | 超過 1000 USDT 需二次確認 |

**運營應用：**
- 活動的自動領取 Agent，只限「領取」，不能動主錢包
- 對外清楚說明 Agent 的權限範圍 → 建立用戶信任
- UI 清楚展示授權內容，避免用戶因不安心而放棄

## 十二、Merkle Tree 與 Merkle Proof

### Merkle Tree 結構
把大量資料逐層雜湊，最終得到一個 Root：

```
            Root
           /    \
        H(AB)  H(CD)
        /  \    /  \
      H(A) H(B) H(C) H(D)
       |    |    |    |
       A    B    C    D
```

- 只要 Root 被可信記錄下來，就能驗証任何一筆資料

### Merkle Proof：証明「我的資料在這批裡」

以証明 A 在名單內為例：

```
提交：A 的原始資料 + Proof：[ H(B), H(CD) ]

合約計算：
  H(A)
  H(A) + H(B) → H(AB)
  H(AB) + H(CD) → Root（算出來的）

比對：算出的 Root == 鏈上的 Root？
✅ 一致 → 証明 A 在這批資料裡
```

- 100 萬筆資料，証明只需約 20 個雜湊值
- Proof 給的是雜湊值，無法反推其他人的原始資料（隱私安全）

### 空投名單（Airdrop List）

空投 ＝ 專案方把代幣免費發給特定錢包地址（獎勵早期用戶、吸引新用戶）

**Merkle Tree 解決的問題：** 10 萬個地址全存上鏈費用極高

```
專案方：
  把「地址 + 可領金額」建成 Merkle Tree
  只把 Root 存上鏈（一個雜湊值，極便宜）

用戶領取：
  提交：地址 + 金額 + Merkle Proof
  合約驗証 → 通過才放行領取
```

**運營應用：**

| 場景 | Merkle Tree 的角色 |
|------|-------------------|
| 空投活動 | 大量名單低成本驗証 |
| 白名單搶購 | 証明有資格參與 Launchpad |
| 積分兌換 | 鏈下積分上鏈時的完整性驗証 |
| 活動任務核實 | 完成任務的用戶名單可驗証 |

## 運營應用

這些概念在 CEX 運營中可以這樣用：
- 私鑰 / 助記詞風險 → 設計資安教育活動（為何要開 2FA、不給私鑰助記詞）
- 數位簽章 → 鏈上活動授權 UI 文案設計，避免用戶盲簽
- 雜湊串鏈 → 對用戶解釋「交易紀錄不可造假」，建立信任感
- 公鑰 / 地址 → 解釋「為何公開地址沒有風險」
- 多簽 → 展示平台資產保管透明度，強化信任
- 帳戶抽象 → 設計無摩擦活動流程，降低新用戶上手門檻
- Merkle Tree → 空投 / 白名單活動的技術基礎，設計清楚的查詢 UI
