## 核心問題

你在 Web3 裡的「身份」到底是怎麼來的？

---

## 核心概念一：私鑰（Private Key）

**本質：** 一個非常大的隨機數字。

0x4c0883a69102937d6231471b5dbb6e538ead2792b820b55b3c3b1d6e...

- 以太坊私鑰是 256 位元的隨機數  
- 沒有任何特殊含義，就是一串隨機產生的數字

**為什麼必須是「真正隨機」？**

必須使用**密碼學安全的隨機數產生器（CSPRNG，Cryptographically Secure Random Number Generator）**，不能用一般的偽隨機數。

Q：如果私鑰可以被預測，我可以透過公鑰來回推私鑰嗎？ A：「可以被預測」跟「從公鑰反推」是兩件不同的事。如果私鑰產生器很爛（例如只產生 1、2、3……），攻擊者不需要你的公鑰，直接暴力掃遍所有可能的私鑰，找到有錢的地址就盜走。

**兩種攻擊路徑對比：**

| 攻擊方式 | 防禦機制 |
| :---- | :---- |
| 從公鑰反推私鑰 | 橢圓曲線的單向性 |
| 暴力猜測所有可能的私鑰 | 256 位元真隨機，可能性多到猜不完（2²⁵⁶ ≈ 10⁷⁷，比宇宙所有原子數量還多） |

## 核心概念二：從私鑰到地址的完整推導鏈

私鑰（256位元隨機數）

      ↓ 橢圓曲線運算（secp256k1）

公鑰（64 bytes，128個字符）

      ↓ Keccak-256 Hash，取後 20 bytes

地址（0x 開頭，40個字符）

每一步都是**單向的**：

- 可以從私鑰推到地址  
- 沒辦法從地址反推回公鑰  
- 更沒辦法從公鑰反推回私鑰

**為什麼不直接用公鑰當地址？**

原始公鑰有 128 個字符，地址只有 40 個字符，有兩個好處：

1. **方便使用**：複製貼上比較不容易出錯  
2. **多一層安全保護**：從地址沒辦法反推出公鑰，對未來量子電腦攻擊多一層防護

Q：地址 40 個字符有比較實際嗎？人也不會真的記住。 A：對，沒有人真的在「記」地址，都是複製貼上。地址的設計目標是「比公鑰短、機器好處理、複製貼上不容易出錯」，不是讓人背。實際使用上可以用 ENS（Ethereum Name Service）把地址變成 `jane.eth` 這樣的人類可讀格式。

**真實風險：Address Poisoning（地址投毒）**

因為沒人真的看完整個地址，駭客會製造開頭和結尾跟你一樣的假地址：

你的真實地址：0x742d...f44e

駭客的假地址：0x742d...a31f  ← 開頭一樣，結尾不同

→ 轉帳前至少要比對地址的**開頭和結尾幾個字符**。

## 核心概念三：助記詞（Mnemonic Phrase）

**本質：** 私鑰的人類可讀版本。

私鑰是一串亂碼，沒辦法記。**BIP-39** 標準把這個隨機數轉換成 12 或 24 個英文單字：

witch collapse practice feed shame open

despair creek road again ice least

助記詞和私鑰之間有**固定的數學對應關係**：

- 給你助記詞 → 可以還原出私鑰  
- 給你私鑰 → 可以推回助記詞

**最重要的觀念：助記詞 \= 私鑰**

助記詞不是「密碼」，它就是私鑰本身的另一種表達形式。

| 私鑰的規則 | 助記詞同樣適用 |
| :---- | :---- |
| 私鑰不能給別人 | 助記詞也不能給別人 |
| 私鑰丟了就沒了 | 助記詞丟了也一樣沒了 |
| 有人要你給私鑰 \= 釣魚 | 有人要你輸入助記詞 \= 釣魚 |

**備份原則：**

- 抄在紙上，離線保存  
- 不要截圖、不要存雲端、不要用任何 App 記錄  
- 任何合法服務都不需要你的助記詞

## 核心概念四：HD 錢包（Hierarchical Deterministic Wallet）

**問題：** 如果你有很多個錢包地址，要備份很多組助記詞嗎？

**解法：** 一組助記詞管理所有地址。

**直覺：種子長出一棵樹**

助記詞是一顆種子，從這顆種子長出一棵樹，樹上每一根樹枝就是一個錢包地址：

助記詞（種子）

      ↓

  主私鑰

  /    \\

地址1  地址2

       /    \\

    地址3  地址4

標準：**BIP-32/44**，推導路徑：

助記詞

  ↓ BIP-39

種子（Seed）

  ↓ BIP-32

主私鑰（Master Key）

  ↓ BIP-44

m / 44' / 60' / 0' / 0 / 0  → 以太坊第一個地址

m / 44' / 60' / 0' / 0 / 1  → 以太坊第二個地址

m / 44' / 60' / 0' / 0 / 2  → 以太坊第三個地址

（路徑不需要記，錢包軟體自動處理）

**實際意義：**

- 在 MetaMask 裡「新增帳戶」新增多個地址，只需備份一組助記詞  
- 換手機或重裝錢包，輸入同一組助記詞，所有地址全部回來

---

## 完整推導鏈總覽

真隨機數（CSPRNG）

        ↓

私鑰（256位元隨機數）

        ↓ 橢圓曲線運算（secp256k1）

公鑰（64 bytes）

        ↓ Keccak-256 Hash，取後 20 bytes

地址（0x開頭，40字符）

備份方式：

私鑰 ↔ 助記詞（12個單字，BIP-39）

一組助記詞管理多個地址：

助記詞 → 種子 → 主私鑰 → 無限多個地址（BIP-32/44）

## 關鍵問答

Q：朋友買了新手機，想把舊手機的錢包搬過去。他問：「我需要把每個地址的私鑰都抄下來嗎？」 A：不用，只需要記住助記詞就好。輸入同一組助記詞，所有地址全部回來。而且助記詞要抄在紙上離線保存，不要截圖、不要存雲端。

## 與 Agent Payment 的連結

Agent 付款需要持有私鑰才能簽名。但你不可能把助記詞或私鑰直接交給 AI agent——那等於把所有資產的控制權交出去。這就是為什麼 Skyfire 這類方案需要更精細的權限控制，而不是直接給 agent 一把完整的私鑰。解法在第 17 章帳戶抽象的 **session key** 概念裡。
