## 核心問題

當錢包彈出一個簽名請求，你怎麼判斷該不該按下去？

---

## 核心概念一：兩種簽名請求的差異

**情境 A（危險）：**

Sign Message

0x4a7b3c2d1e9f8a6b5c4d3e2f1a0b9c8d7e6f5a4b3c2d1e0f...

（一串看不懂的亂碼）

**情境 B（安全）：**

Sign In to OpenSea

domain: opensea.io

address: 0x742d...f44e

statement: I accept the OpenSea Terms of Service

nonce: 8f7e6d

issued-at: 2026-06-08T10:30:00Z

**關鍵差異：你能不能看懂你在簽什麼。**

---

## 核心概念二：EIP-712（結構化簽名）

**設計目標：** 讓你在簽名之前，能用人類看得懂的語言確認你在授權什麼。

每個欄位各自防禦一種攻擊：

| 欄位 | 用途 | 防禦的攻擊 |
| :---- | :---- | :---- |
| `domain` | 標明是哪個網站發出的請求 | 釣魚網站冒充合法網站 |
| `address` | 確認是哪個地址在簽名 | 確認是本人操作 |
| `statement` | 讓你看懂你在同意什麼 | 隱藏授權內容 |
| `nonce` | 一次性隨機數，用過就作廢 | Replay Attack（重放攻擊） |
| `issued-at` | 簽名的時間戳記 | 簽名過期後無法使用 |

**domain 的重要性：** EIP-712 把 domain 鎖進簽名裡，這個簽名只對這個網域有效。釣魚網站 `0pensea.io` 拿到的簽名，在真正的 `opensea.io` 上用不了。

---

## 核心概念三：nonce（一次性隨機數）

Q：nonce 是什麼？跟第 2 章礦工找的 Nonce 一樣嗎？ A：完全不同的東西，只是同一個詞。這裡的 nonce 是「一次性隨機數」，目的是防止 Replay Attack。

**Replay Attack（重放攻擊）是什麼：** 攻擊者把你過去用過的簽名存起來，拿去重複使用。

你簽名登入 → 登出

      ↓

攻擊者拿你的舊簽名去伺服器說：「讓我用她的帳號登入」

      ↓

伺服器檢查 nonce → 已使用 → 拒絕

**nonce 怎麼產生：**

Q：nonce 是誰產生的？ A：伺服器產生，不是你、也不是你的錢包。

你點「登入」

      ↓

伺服器產生隨機 nonce（例如 8f7e6d）

      ↓

把整包訊息送到你的錢包

      ↓

你簽名，nonce 被鎖進簽名裡

      ↓

伺服器驗證簽名，把這個 nonce 標記為「已使用」

      ↓

下次登入，伺服器產生全新的 nonce

為什麼必須由伺服器產生？因為需要有人負責記錄「這個 nonce 有沒有被用過」，如果讓錢包自己產生，伺服器無法追蹤，防禦就失效了。

一句話總結：nonce 是伺服器出的考題，你用私鑰簽名作答，答案用過一次就作廢。

**nonce 跟私鑰是兩道不同的鎖：**

| 鎖 | 防禦的事 |
| :---- | :---- |
| nonce | 防止舊簽名被重複使用 |
| 私鑰的單向性 | 防止偽造新簽名 |

Q：駭客怎麼複製我的簽名？ A：區塊鏈上所有廣播出去的資料都是公開的，駭客直接從鏈上或網路封包抓就有了。複製簽名根本不需要駭客技術——這正是為什麼 nonce 機制這麼重要。

## 核心概念四：SIWE（Sign-In with Ethereum，用以太坊登入）

**傳統登入的問題：**

- 帳號密碼存在網站伺服器，網站被駭就洩漏  
- 「用 Google 登入」還是把身份交給 Google 管理  
- 網站收集你的個人資料

**SIWE 的邏輯：** 不需要帳號密碼，只需要證明「你持有某個錢包地址的私鑰」。

你點「用錢包登入」

      ↓

伺服器產生訊息（含 domain、nonce 等）

      ↓

你的錢包彈出視窗，確認後簽名

      ↓

伺服器用你的公鑰驗證簽名 ✓

      ↓

登入成功

網站不需要知道你的名字、email、住址——只知道「這個人持有這個地址的私鑰」。

**三種登入方式對比：**

|  | 傳統帳密 | 用 Google 登入 | SIWE |
| :---- | :---- | :---- | :---- |
| 身份存在哪 | 網站伺服器 | Google | 你自己的錢包 |
| 洩漏風險 | 網站被駭就洩漏 | Google 被駭就洩漏 | 只要私鑰安全就沒事 |
| 個人資料 | 網站拿走 | Google 拿走 | 網站什麼都不知道 |
| 忘記密碼 | 要重設 | 要找 Google | 不存在這個問題 |

現實應用：很多 NFT 平台、DeFi 協議連上錢包後要你簽的「登入訊息」，就是 SIWE。

## 核心概念五：釣魚簽名怎麼騙人

**Malicious Approval（惡意授權）——最常見的盜竊手法：**

釣魚網站彈出的簽名請求：

0x095ea7b3000000000000000000000000\[某個地址\]

ffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffffff

翻譯成人話：

0x095ea7b3    → ERC-20 的 approve() 函數（授權別人動你的代幣）

\[某個地址\]    → 授權這個地址

ffffffff...   → 動用你「無限量」的代幣

你按下去的瞬間，對方隨時可以把你的代幣全部轉走。

## 如何判斷一個簽名請求安不安全

**核心原則：看不懂你在簽什麼，就不要簽。**

檢查清單：

| 檢查項目 | 安全 | 危險 |
| :---- | :---- | :---- |
| 內容可讀性 | 人類看得懂的文字 | 一串亂碼或編碼 |
| domain | 跟你連上的網站一致 | 不明網域或對不上 |
| 授權範圍 | 明確限定的操作和金額 | `unlimited` 或 `ffffffff...` |
| 來源 | 你主動操作觸發的 | 突然彈出來、來路不明 |

**關鍵問答：** Q：dApp 要求簽名「Approve unlimited USDC spending」，可以簽嗎？ A：不要簽。任何正常的網站都不應該向使用者要求無上限授權。如果真的需要授權，要求它只授權這次交易需要的金額（例如 100 USDC）就好。

**實際操作：你怎麼簽名？**

你不需要手動做任何事，錢包軟體會幫你全部處理好：

伺服器送來簽名請求

      ↓

錢包自動彈出視窗

      ↓

你看清楚內容後按下「Sign」

      ↓

錢包在背後自動：訊息跑 Hash → 私鑰做橢圓曲線運算 → 產生數位簽名

      ↓

簽名自動送回伺服器

你唯一需要做的：**看清楚視窗裡寫了什麼，再按下按鈕。** 按下去就無法撤銷。

## 與 Agent Payment 的連結

Agent 自動付款時，每一筆付款本質上都是一個簽名動作。這就是為什麼 agent 的簽名權限必須被嚴格限制——不能給它簽「無限授權」，只能讓它在特定條件下簽特定金額。這正是第 17 章 **session key** 要解決的核心問題。

SIWE 的概念延伸到 agent 身上，就是「agent 怎麼向服務提供商證明自己的身份」——不是用帳號密碼，而是用簽名。Skyfire 這類方案讓 agent 持有一個身份，每次付款或呼叫 API 時，用簽名證明「這個請求是被授權的 agent 發出的」。
