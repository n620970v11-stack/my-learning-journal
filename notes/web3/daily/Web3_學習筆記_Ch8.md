## 核心問題

合約寫好了，接下來怎麼測試、部署、讓前端跟它互動？

---

## 核心概念一：三個環境的分工

開發智能合約跟開發一般產品一樣，正式上線前要經過三個環境：

```
本地測試鏈（Local）→ 測試網（Testnet）→ 主網（Mainnet）
     內部開發            staging            正式上線
```

### 本地測試鏈（Local Testnet）
- 跑在自己電腦上的「假以太坊」
- 完全免費，測試幣要多少有多少
- 只有自己看得到，重開就清空
- 適合開發階段快速試錯
- 常用工具：**Hardhat Network**、**Anvil**（Foundry 內建）

### 測試網（Testnet）
- 真實的以太坊網路，但用沒有真實價值的測試幣
- 跟主網行為幾乎一樣
- 測試幣從「水龍頭（Faucet）」免費領取
- 適合上線前的最後驗證
- 目前以太坊主要測試網：**Sepolia**

### 主網（Mainnet）
- 真實的以太坊，用真實的 ETH 付 Gas
- 合約一旦部署就永久存在、改不掉
- 每一步都有真實成本
- 只有充分測試過才應該部署到這裡

> **Agent Payment 連結：** x402、Skyfire 這類方案開發期都跑在測試網。如果文件說「目前支援 Sepolia testnet」，代表它還在測試階段，尚未準備好讓真實資金流動。評估一個 agent payment 方案的成熟度，看它有沒有主網部署是重要指標之一。

## 核心概念二：Hardhat vs Foundry

兩個框架解決同一個問題：**幫開發者編譯合約、跑測試、部署到鏈上。**

| | Hardhat | Foundry |
|--|---------|---------|
| 測試語言 | JavaScript / TypeScript | Solidity |
| 速度 | 較慢 | 極快 |
| 跟前端整合 | 非常好 | 需要額外設定 |
| 適合誰 | 前端背景的開發者 | 合約重度開發者 |
| 生態成熟度 | 成熟、教學資源多 | 近年快速崛起 |

**PM 的判斷點：**
- 工程師說「我們用 Foundry」→ 他們很認真在做合約測試，這是好事
- 一個合約專案完全沒有測試框架 → 這是警訊

## 核心概念三：前端怎麼跟鏈互動

### 基本架構

前端不能直接跟區塊鏈說話，中間需要一個**翻譯層**：

```
你點「Swap」按鈕
      ↓
前端（ethers.js / viem）← 翻譯層
      ↓
以太坊節點
      ↓
智能合約執行
      ↓
區塊鏈狀態更新
```

> 類比：就像網頁前端不能直接改資料庫，需要透過 API 跟後端溝通。以太坊的「後端 + 資料庫」換成了「智能合約 + 區塊鏈」。

### 三個工具的關係

| 工具 | 角色 | 類比 |
|------|------|------|
| ethers.js | 前端跟鏈溝通的翻譯層（老牌） | 直接操作引擎 |
| viem | 前端跟鏈溝通的翻譯層（新世代，更快） | 更好的引擎 |
| wagmi | 錢包連線 + 狀態管理（建在 viem 上） | 方向盤 + 儀表板 |

```
wagmi        ← 處理錢包連線、帳戶狀態管理
  └── viem   ← 處理跟鏈的溝通
        └── 以太坊節點
```

目前業界趨勢：新專案傾向用 **viem + wagmi** 的組合；ethers.js 是老牌，許多舊專案仍在使用。

## 核心概念四：三個 Debug 與監控工具

### Etherscan — 鏈上的「Google」
- **區塊瀏覽器（Block Explorer）**，完全公開，不需要登入
- 可以查：地址的所有交易紀錄、交易詳細內容、合約程式碼
- **PM 用途：** 用戶說付款沒反應，把交易 hash 貼上去，三秒內知道交易成功、失敗、還是 pending

### Tenderly — 合約的「偵錯器」
- 把一筆交易拆解成每個執行步驟，找出在哪一行出錯
- Etherscan 告訴你「交易失敗了」，Tenderly 告訴你「**為什麼**失敗」

```
步驟 1：呼叫 transfer() ✅
步驟 2：檢查餘額 ✅
步驟 3：呼叫外部合約 ❌ ← 在這裡 revert，原因是「餘額不足」
```

> 類比：Etherscan 是驗屍報告（告訴你人死了），Tenderly 是解剖室（告訴你怎麼死的）。
> **PM 用途：** 跟工程師開 bug review 時，能看懂 Tenderly 的報告，不需要完全靠工程師翻譯。

### The Graph — 鏈上資料的「查詢引擎」
- 區塊鏈的資料是「一個區塊接一個區塊」儲存的，不是關聯式資料庫，無法直接下 SQL query
- The Graph 把鏈上事件資料「索引」起來，讓你用 GraphQL 查詢
- **PM 用途：** 鏈上數據分析、監聽特定事件（例如每次有人用 agent 付款就觸發通知）

> 類比：區塊鏈是一本流水帳，The Graph 幫你把它建成有索引的資料庫。

### 三個工具對比

| 工具 | 解決什麼問題 | PM 使用情境 |
|------|------------|------------|
| Etherscan | 查交易狀態、地址歷史 | 用戶回報問題時快速確認 |
| Tenderly | 合約執行細節、debug | 跟工程師討論 bug |
| The Graph | 鏈上資料查詢、事件監聽 | 數據分析、產品監控 |

## ✓ 學完能做到（PM 版本）

聽到「合約在 Sepolia 上 revert 了，我要用 Tenderly debug」這句話，能理解：
- **Sepolia** = 測試網，還在測試階段，用假的測試幣
- **revert** = 交易已經跑了，但執行到一半出錯、被強制中止並回滾
- **Tenderly debug** = 要把這筆失敗的交易拆解成每個步驟，找出哪行程式碼出問題

---

## 與 Agent Payment 的連結

- **三個環境**：agent payment 方案從本地測試 → Sepolia 測試網 → 主網，是標準的開發流程。主網部署代表產品成熟，可以流動真實資金。
- **The Graph**：監聽 agent 付款事件的標準工具，可以知道「哪個 agent、什麼時候、付了多少給誰」。
- **Tenderly**：agent payment 合約出問題時，這是工程師找原因的第一個工具。
