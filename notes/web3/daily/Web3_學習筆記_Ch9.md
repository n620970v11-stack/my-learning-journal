# Web3 學習筆記｜第 9 章：代幣標準

> 學習目標：自己發一個代幣、自己鑄一個 NFT，並理解標準介面為什麼長那樣
> 完成日期：2026/06

---

## 核心問題

以太坊上有各種各樣的代幣，大家怎麼確保它們能互相相容、能被錢包和交易所認識？

---

## 核心概念一：為什麼需要標準

沒有標準的世界：
- A 公司的代幣，轉帳功能叫 `send()`
- B 公司的代幣，轉帳功能叫 `transfer()`
- C 公司的代幣，轉帳功能叫 `move()`

每個錢包、每個交易所都要為每種代幣寫特別的程式碼——根本無法擴展。

**有了標準之後：**
所有人都用 `transfer()`，錢包只需要寫一次程式碼，就能支援所有遵守標準的代幣。

### ERC 是什麼

**ERC（Ethereum Request for Comments）** 是以太坊社群提出和討論標準的機制。任何人都可以提案，經過社群討論和審查後成為正式標準。

> 類比：ERC 像**建築法規**。法規不蓋房子，但規定「所有房子必須有逃生出口、門的寬度至少多少」。開發者按照規格蓋合約，所有工具（錢包、交易所）就能認識它。

**重要觀念：ERC 是「最低要求」，不是「全部規定」**

ERC 規定合約**至少要有**哪些功能，但你可以在基礎上加更多。只要包含標準規定的功能，錢包和交易所就認識你。

```
ERC 標準（最低要求）
      +
開發者自己加的額外功能
      ↓
最終的代幣合約
```

例如：USDC 除了 ERC-20 基本功能，還額外加了「凍結帳戶」功能（Circle 可以凍結被懷疑洗錢的地址）。

### ERC 的分類框架

| 類別 | 標準 | 解決什麼 |
|------|------|---------|
| 資產類型 | ERC-20 | 同質化代幣 |
| 資產類型 | ERC-721 | 非同質化代幣 |
| 資產類型 | ERC-1155 | 混合型（多種資產同一合約） |
| 資產行為 | ERC-4626 | 金庫標準（存入生息） |
| 帳戶與互動 | ERC-4337 | 帳戶抽象（第 17 章深入） |
| 帳戶與互動 | EIP-712 | 結構化簽名（第 5 章學過） |
| 安全與授權 | ERC-2612 | Permit（簽名授權，省去 approve 交易） |

## 核心概念二：同質化 vs 非同質化

**同質化（Fungible）**：每一個單位都完全一樣、可以互換。
- 例子：錢。你的 100 元跟我的 100 元完全等價，可以互換。

**非同質化（Non-Fungible）**：每一個都獨一無二，不可互換。
- 例子：畢卡索的畫。每一幅都是不同的東西，不能直接互換。

| | 同質化 | 非同質化 |
|--|--------|---------|
| 例子 | ETH、USDC、治理代幣 | NFT、藝術品、門票 |
| 特性 | 每單位完全相同 | 每個都獨一無二 |
| 對應標準 | ERC-20 | ERC-721 |

## 核心概念三：ERC-20（同質化代幣標準）

### 六個核心功能

```
transfer()      → 把代幣轉給別人
approve()       → 授權某個地址幫你花代幣
transferFrom()  → 被授權的地址幫你轉代幣
balanceOf()     → 查某個地址有多少代幣
allowance()     → 查某個地址被授權能花多少
totalSupply()   → 查這個代幣的總發行量
```

### approve() + transferFrom() 的授權機制

這組機制解決「如何讓第三方合約在限定範圍內動用你的代幣」：

```
第一步：你呼叫 approve()
「我授權 Uniswap 合約，最多可以動用我的 100 USDC」

第二步：Uniswap 合約呼叫 transferFrom()
「我要動用剛才被授權的 100 USDC，把它轉走」
```

> 類比：你給了 Uniswap 一張「限額支票」——它只能在你授權的金額範圍內動用你的錢，不能多拿。

**Agent Payment 的直接應用：** agent 替你付款，本質上就是你先 `approve()` 給某個合約，讓它在限定範圍內幫你花錢。

## 核心概念四：ERC-721（非同質化代幣標準）

每一個 ERC-721 代幣都有一個**唯一的 ID**，這個 ID 是它跟其他所有代幣的區別。

```
ERC-20：我有 100 USDC（一個數量）
ERC-721：我有 ID #4521 這個 NFT（一個獨特的個體）
```

### 核心功能

```
ownerOf(tokenId)    → 查 ID #xxx 這個 NFT 現在是誰的
transferFrom()      → 把某個 NFT 轉給別人
approve()           → 授權某個地址可以動這個 NFT
balanceOf(address)  → 查某個地址持有幾個 NFT（只有數量）
```

### NFT 實際存什麼

NFT 本身**不存圖片**，存的是一個指向圖片的連結（metadata）：

```
NFT #4521 合約裡存的：
tokenURI → "https://ipfs.io/ipfs/Qm...abc"

這個連結指向一個 JSON 檔案：
{
  "name": "Bored Ape #4521",
  "image": "https://ipfs.io/ipfs/Qm...xyz"  ← 圖片在這裡
}
```

圖片存在 **IPFS** 或 **Arweave**（去中心化儲存網路）上，不在鏈上——因為把圖片直接存鏈上會貴到不可思議。

> 「NFT 只是一個連結」技術上沒錯，但重點是**所有權紀錄存在鏈上、不可篡改**。

### NFT 的應用範圍

| 應用 | 說明 |
|------|------|
| 數位藝術 | 最常見的用法 |
| 活動門票 | 每張票有唯一 ID，防偽又可轉讓 |
| 會員資格 | 持有某個 NFT 才能進某個社群 |
| 遊戲道具 | 某把劍、某塊土地 |
| 現實資產憑證 | 房產、版權（還在發展中） |

## 核心概念五：ERC-1155（混合型標準）

### 解決什麼問題

開發一款區塊鏈遊戲，有同質化道具（金幣、普通寶劍）也有非同質化道具（傳說寶劍 #001）。

用舊標準：需要部署多個合約，費用高、管理麻煩。

**ERC-1155：一個合約，管所有類型的代幣。**

```
一個 ERC-1155 合約裡可以同時有：
ID #1：遊戲金幣（同質化，可以有很多枚）
ID #2：普通寶劍（同質化，可以有很多把）
ID #3：傳說級寶劍 #001（非同質化，只有一個）
```

> 類比：ERC-20 是只賣一種飲料的販賣機，ERC-721 是只賣獨特收藏品的販賣機，ERC-1155 是**便利商店**——什麼都能賣，統一管理。

### 批次操作省 Gas

ERC-1155 可以一次轉多種代幣，只付一次 Gas：

```
一筆交易：
100 枚金幣 + 5 把普通寶劍 + 傳說寶劍 #001
全部一起轉給另一個地址 → 只付一次 Gas
```

## 核心概念六：ERC-4626（金庫標準）

### 類比：鏈上定存

```
定存：
你把 10,000 元放進銀行 → 銀行給你定存憑證
銀行拿去放貸產生利息 → 一年後你領回 10,300 元

ERC-4626：
你把 10,000 USDC 存進金庫合約 → 合約給你「份額代幣（Shares）」作為憑證
金庫拿去做 DeFi 策略產生收益 → 你把份額代幣還回去，領回 10,300 USDC
```

**份額代幣**本身是 ERC-20 代幣，可以轉讓、可以交易。

### 為什麼需要這個標準

ERC-4626 出現前，每個 DeFi 協議的金庫介面都不一樣，開發者每次整合都要重新學。

ERC-4626 統一介面：

```
deposit()     → 存入資產，取得份額
withdraw()    → 用份額領回資產
totalAssets() → 查金庫現在管理多少資產
```

---

## 三個資產類型標準對比

| 標準 | 特性 | 適合用途 |
|------|------|---------|
| ERC-20 | 同質化，每單位相同 | 穩定幣、治理代幣 |
| ERC-721 | 非同質化，每個獨一無二 | NFT、藝術品、門票 |
| ERC-1155 | 混合型，一個合約管多種 | 遊戲道具、多資產應用 |

## ✓ 學完能做到

看到這段白皮書描述，能完整解讀：

> 「Agent 的預算以 ERC-20 穩定幣計價，存入 ERC-4626 金庫管理，每次付款授權透過 approve() 機制執行。」

**解讀：**
Agent 使用現有的 ERC-20 穩定幣（如 USDC）作為支付單位，這筆預算存進按照 ERC-4626 標準建立的金庫裡統一管理（在等待期間自動生息）。每次 agent 需要付款時，透過 ERC-20 的 `approve()` 機制授權金庫合約動用指定金額，執行付款。

---

## 與 Agent Payment 的連結

- **ERC-20 的 approve() 機制**：agent 替你付款的底層授權方式，你先授權合約能動用多少，agent 再在範圍內執行。
- **ERC-4626 金庫**：agent 的預算不只是「鎖住等著被花」，而是存進金庫在等待期間持續生息——這是加密原生 agent payment 比傳統預付卡更有效率的地方。
- **ERC-1155 批次操作**：agent 如果需要一次處理多種代幣的付款，ERC-1155 可以大幅降低 Gas 成本。
