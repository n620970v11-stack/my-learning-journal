## 核心問題

智能合約到底是什麼？它存在哪裡？怎麼運作？

## 核心概念一：智能合約存在哪裡

智能合約的程式碼被永久寫進區塊鏈上，全網幾千台電腦每一台都有一份完整的副本。

**部署合約的流程：**

```
開發者用 Solidity 寫程式碼
      ↓
發出一筆「部署交易」（收件人是空白）
      ↓
廣播給全網 → 進入 Mempool
      ↓
礦工／驗證者打包進區塊（PoW 或 PoS 確認）
      ↓
以太坊自動分配一個合約地址
      ↓
全網幾千台電腦同步這個區塊，都有這份程式碼的副本
```

> Q：程式碼上鏈也需要礦工執行嗎？
> A：對！部署合約本質上就是一筆特殊的交易。普通交易的「收件人」是某個地址，部署合約的交易「收件人是空白」——這就是以太坊識別「這是部署合約」的方式。流程跟普通交易完全一樣，需要付 Gas，而且通常比普通轉帳貴很多（程式碼越複雜費用越高）。

## 核心概念二：EVM（Ethereum Virtual Machine，以太坊虛擬機）

**EVM 是什麼：**
一個**標準化的執行環境**，安裝在全網每一台電腦上，負責執行智能合約的程式碼。

```
合約程式碼
      ↓
EVM（每台電腦上都有一份）
      ↓
執行結果在全網完全一致
```

**為什麼需要 EVM：**
全網幾千台電腦，硬體和作業系統都不一樣。如果直接在各自的電腦上執行合約，結果可能不一致——共識就崩潰了。

> Q：如果執行結果不一致會怎樣？
> A：會起爭執——共識崩潰。5,000 台電腦算出結果 A，另外 5,000 台算出結果 B，帳本不一致，區塊鏈分裂成兩條，整個網路失去信任基礎。

**類比：**
就像 Python 的執行環境——不管你用 Windows、Mac、Linux，Python 程式跑出來的結果都一樣。EVM 對智能合約做的事情一樣。

> Q：EVM 怎麼安裝？
> A：你不需要自己安裝。EVM 內建在驗證者節點軟體裡（例如 Geth、Nethermind）。你作為使用者，只需要發出交易，驗證者的 EVM 幫你執行合約。開發者測試合約時，會用 Hardhat 或 Foundry 在本地跑一個模擬的 EVM。

**全網同步的機制：P2P 網路 + Gossip Protocol**
```
驗證者產生新區塊
      ↓
廣播給直接連線的幾十個節點
      ↓
這些節點驗證後再廣播給各自連線的節點
      ↓
像八卦消息一樣，幾秒內傳遍全網
```

每個節點收到新區塊都會獨立驗證（Hash 合法？簽名正確？沒有雙重花費？），不是盲目跟隨。

## 核心概念三：以太坊網路的角色

| 角色 | 做什麼 |
|------|--------|
| 使用者 | 發出交易、使用 dApp |
| 驗證者／礦工 | 打包交易、驗證區塊、維護網路 |
| 全節點（Full Node）| 下載完整區塊鏈、獨立驗證所有交易 |
| 輕節點（Light Node）| 只下載區塊 Header，不驗證全部（手機錢包）|

理論上任何人都可以成為驗證者——只要質押 32 ETH，裝好軟體，就加入網路開始賺取獎勵。

## 核心概念四：Solidity 基本語法

**最簡單的合約範例：**

```solidity
// SPDX-License-Identifier: MIT
pragma solidity ^0.8.0;

contract SimpleStorage {
    uint256 public storedNumber;

    function store(uint256 number) public {
        storedNumber = number;
    }

    function retrieve() public view returns (uint256) {
        return storedNumber;
    }
}
```

**逐行說明：**

| 程式碼 | 意思 |
|--------|------|
| `uint256` | 256 位元的正整數型別 |
| `public` | 任何人都可以讀取／呼叫 |
| `storedNumber` | 永久存在鏈上的變數（storage）|
| `function store()` | 寫入資料，需要付 Gas |
| `view` | 只讀取資料，不修改狀態 |
| `pure` | 完全不碰 storage，只做計算 |
| `returns (uint256)` | 回傳一個 uint256 型別的值 |

**需不需要付 Gas：**

| 操作 | 需要付 Gas 嗎 | 原因 |
|------|-------------|------|
| 寫入資料（store）| ✅ 需要 | 改變鏈上狀態 |
| 讀取資料（retrieve，view）| ❌ 不需要 | 只查詢，不改變任何東西 |
| 純計算（pure）| ❌ 不需要 | 完全不碰鏈上資料 |

**智能合約 vs 傳統資料庫（SQL）：**

| | 傳統資料庫（SQL）| 智能合約（Solidity）|
|--|----------------|-------------------|
| 費用 | 付給資料庫服務商 | 付 Gas 給驗證者 |
| 控制權 | 管理員可以修改、刪除資料 | 部署後沒有人能修改 |
| 信任基礎 | 信任資料庫服務商 | 信任程式碼邏輯 |
| 透明度 | 通常不公開 | 任何人都能看到 |
| 停機風險 | 服務商倒閉就沒了 | 網路存在就一直跑 |

> Q：我為什麼要寫合約？我的好處是什麼？
> A：分兩個角色：
> - **使用者**：不需要寫合約，只是使用別人部署的合約（Uniswap、Aave 等），付 Gas 是使用服務的費用。
> - **開發者**：寫合約是為了建立服務——不需要伺服器、規則無法被修改、可以自動收取手續費獲利。
> - **你的目標**：學 Solidity 是為了看懂別人寫的合約，評估 DeFi 協議的邏輯和風險，以及理解 agent payment 合約的設計。

## 核心概念五：Storage / Memory / Calldata

**類比：**
- **storage**：公司的檔案室——永久存放，費用高
- **memory**：桌上的便條紙——只在這次用，用完就丟
- **calldata**：別人交給你的文件——只能看，不能修改

**三種資料位置對比：**

| 位置 | 存在哪裡 | 生命週期 | Gas 費用 | 例子 |
|------|---------|---------|---------|------|
| storage | 鏈上永久儲存 | 永久 | 最貴 | `storedNumber` |
| memory | 執行期間的臨時空間 | function 結束就消失 | 便宜 | function 內的臨時變數 |
| calldata | 外部傳進來的輸入 | 只能讀不能改 | 最便宜 | function 的參數 |

**範例：**
```solidity
function double(uint256 number) public pure returns (uint256) {
    uint256 result = number * 2;  // result 在 memory，用完就消失
    return result;
}
```

> Q：`result` 存在哪裡？
> A：memory。它是 function 執行期間產生的臨時變數，用完就消失，不會寫進鏈上。

## 核心概念六：合約之間的互動

真實世界的 DeFi 不是只有一個合約，合約之間會互相呼叫。

**三種互動方式：**

| 方式 | 類比 | 在誰的環境執行 | 改變誰的資料 | 能改資料嗎 |
|------|------|-------------|-------------|-----------|
| call | 打電話請甜點店做蛋糕（甜點店用自己的廚房）| 合約 B | 合約 B 的 storage | ✅ 能 |
| delegatecall | 請甜點店來我廚房，用我的材料按他的食譜做 | 合約 A | 合約 A 的 storage | ✅ 能 |
| staticcall | 打電話問甜點店今天有什麼口味（只問不做）| 合約 B | 無 | ❌ 不能 |

**delegatecall 的特殊用途：可升級合約**

合約部署後本來沒辦法修改，但 delegatecall 可以實現升級：
```
使用者 → 呼叫代理合約（Proxy，永遠不變）
              ↓ delegatecall
         邏輯合約（Logic）← 開發者可以換掉這個
```

**delegatecall 的風險：**

> Q：看到合約用了 delegatecall，你會擔心什麼？
> A：可能會把合約 A 的資料捲走。具體風險有兩個：
> 1. **被呼叫的合約是惡意的**：合約 B 的程式碼在合約 A 的環境執行，可以直接操作合約 A 的 storage，把所有資產轉走。
> 2. **合約 B 可以被升級換掉**：今天正常，明天換成惡意版本，合約 A 還是照樣呼叫它。
>
> 看到 delegatecall 要問：被呼叫的合約可信嗎？升級權限在誰手上？

## 與 Agent Payment 的連結

Skyfire 這類 agent payment 方案，核心是一個智能合約定義 agent 的付款權限。這個合約很可能用了 delegatecall 來實現可升級性——但這也代表你需要信任升級權限的持有者不會作惡。評估一個 agent payment 方案的安全性，看它的合約架構是很重要的一步。
