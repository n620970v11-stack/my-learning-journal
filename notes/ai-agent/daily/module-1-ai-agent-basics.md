# 模組一：AI Agent 基礎概念

### Module 1: AI Agent Fundamentals

學習目標：建立 AI Agent 的核心心智模型，為後續 Web3 應用打下基礎。

---

## 💡 心智模型：機器人比喻

Agent 就像一個機器人。

- **LLM** 是它的大腦，負責思考、決策、判斷要用哪個工具  
- **工具（Tools）** 是它的手和腳，執行大腦的指令去呼叫外部服務  
- **Context Window** 是它的工作記憶體（RAM），儲存當下任務的資訊，關機就消失  
- **長期記憶資料庫** 是它的硬碟，跨任務都能記住

**人類記憶 vs Agent 記憶的對照：**

| 類型 | 人類 | Agent |
| :---- | :---- | :---- |
| 短期記憶 | 暫時記在腦中，幾分鐘後可能忘了 | Context Window，對話結束就清空 |
| 長期記憶 | 記得怎麼騎腳踏車、朋友的名字 | 長期記憶資料庫，跨對話都能存取 |

AI 的記憶設計，本來就是參考人類認知科學來設計的。

---

## 核心架構總覽

Agent \= 大腦（LLM）+ 手腳（工具 Tools）+ 記憶體（Memory）

| 組成 | 英文 | 功能 |
| :---- | :---- | :---- |
| 大腦 | LLM | 思考、決策、判斷要用哪個工具 |
| 手腳 | Tools | 執行大腦的指令，呼叫外部 API 或服務 |
| 工作記憶 | Context Window | 短期記憶，儲存當前任務資訊，對話結束即清空 |
| 長期記憶 | Long-term Memory DB | 跨對話都能記住的資訊，儲存於外部資料庫 |

## 1-1 什麼是 AI Agent？

### What is an AI Agent?

**核心差異：主動 vs 被動**

|  | 聊天機器人 Chatbot | AI Agent |
| :---- | :---- | :---- |
| 工作模式 | 一問一答，被動回應 | 主動拆解步驟、執行任務 |
| 知識來源 | 固定資料庫 | 可即時取得外部資訊 |
| 能做的事 | 回答問題 | 執行操作 |
| 遇到新問題 | ❌ 不知道 | ✅ 自己想辦法 |

**一句話總結：**

聊天機器人是**回答問題**的工具；AI Agent 是**完成任務**的夥伴。

**Web3 連結：**

- 聊天機器人：回答用戶「什麼是流動性挖礦？」  
- AI Agent：自動執行流動性挖礦的整個流程——監控利率、選擇池子、執行存入、到期提領

## 1-2 Agent 的核心組成

### Core Components of an Agent

**四個組成：**

| 組成 | 英文 | 說明 | Web3 例子 |
| :---- | :---- | :---- | :---- |
| 感知 | Perception | 接收外部資訊（文字、數據、鏈上資料） | 接收 Uniswap 池子深度、Gas 費、大戶動向 |
| 決策 | Decision-Making | LLM 思考：現在該做什麼？用哪個工具？ | 判斷是否達到止損條件 |
| 行動 | Action | 執行操作（呼叫工具、執行交易） | 簽署鏈上交易、發出通知 |
| 記憶 | Memory | 儲存資訊（短期 \+ 長期） | 記住用戶設定的止損價格與持倉資訊 |

**ReAct Loop（Agent 運作循環）：**

感知（接收資訊）

    ↓

決策（LLM 思考：該做什麼？）

    ↓

行動（執行操作）

    ↓

感知（觀察行動的結果）

    ↓

決策（任務完成了嗎？）

    ↓

… 重複直到目標達成

**ReAct \= Reasoning（推理）+ Acting（行動）**，是現代 Agent 設計的基礎框架。

## 1-3 單一 Agent vs 多 Agent 系統

### Single Agent vs Multi-Agent System

**什麼時候用單一 Agent？**

- 任務明確、步驟不多  
- 可以被一個人獨立完成的事

**什麼時候需要多 Agent？**

| 情況 | 說明 |
| :---- | :---- |
| 任務太複雜 | 超出單一 Agent 的 Context Window 容量 |
| 需要不同專業 | 每個 Agent 有自己的專長和工具 |
| 需要互相檢查 | 一個做、一個審，降低錯誤率 |
| 需要平行處理 | 多個 Agent 同時進行，效率更高 |

**兩種常見結構：**

① 主管式 Orchestrator Pattern

主管 Agent（協調者）

    ├── 子 Agent A（查資料）

    ├── 子 Agent B（分析）

    └── 子 Agent C（輸出報告）

② 接力式 Pipeline Pattern

Agent A（原始資料）→ Agent B（清洗分析）→ Agent C（生成報告）

**Web3 例子（Crypto Exchange）：**

主管 Agent：「用戶想要最佳化收益」

    ├── 市場分析 Agent：掃描各交易對的機會

    ├── 風險評估 Agent：檢查每個機會的風險

    └── 執行 Agent：下單、管理倉位

## 1-4 工具呼叫是什麼？

### Tool Use / Function Calling

**為什麼需要工具？**

LLM 只會「說話」，不會「做事」。工具讓 Agent 突破這個限制。

**工具呼叫流程：**

用戶提出任務

    ↓

LLM 思考：「我需要哪個工具？」

    ↓

呼叫工具（Tool Call）

    ↓

工具回傳結果

    ↓

LLM 繼續思考 \+ 回應用戶

**常見工具類型：**

| 工具類型 | 例子 |
| :---- | :---- |
| 查詢資料 | 即時幣價 API、鏈上數據、新聞搜尋 |
| 執行操作 | 發送通知、下單、轉帳 |
| 計算處理 | 跑程式碼、計算投資報酬率 |
| 呼叫其他服務 | 寄 Email、更新資料庫、呼叫另一個 Agent |

**Web3 場景最重要的工具類型：**

| 工具 | 用途 |
| :---- | :---- |
| 鏈上數據 API | 查錢包餘額、交易紀錄、Gas 費 |
| DEX 價格 API | 查 Uniswap、Curve 等即時價格 |
| 智能合約呼叫 | 直接與區塊鏈上的合約互動（存款、提款、swap） |
| 風險評分 API | 查合約或地址的安全評分 |

⚠️ **重要提醒：** 在 Web3 裡，「呼叫智能合約」可能直接動到真實資金，是權限控制的核心議題。

**術語補充：**

- Tool Use（工具呼叫）= Function Calling（函數呼叫），同一件事的兩種說法。

## 1-5 Agent 的自主性層級

### Levels of Agent Autonomy

**核心概念：**

自主性 \= Agent 完成任務時，需要人類介入的程度。 介入越多 → 自主性越低；介入越少 → 自主性越高。 **不是越高越好**——特別是 Web3 涉及真實資金的場景。

**五個層級：**

| 層級 | 名稱 | 人類角色 | Web3 適用場景 |
| :---- | :---- | :---- | :---- |
| Level 1 | 純輔助 Pure Assistant | 決策者 | 市場資訊查詢 |
| Level 2 | 建議確認 Recommend & Confirm | 審核者 | 交易下單、止損設定 ⭐ 目前主流 |
| Level 3 | 有限自主 Constrained Autonomy | 規則制定者 | DeFi 自動化策略 ⭐ 目前主流 |
| Level 4 | 高度自主 High Autonomy | 監督者 | 機構級投資組合管理 |
| Level 5 | 完全自主 Full Autonomy | 旁觀者 | ⚠️ 目前不建議 |

**PM 的甜蜜點設計原則：**

- 太低 → 用戶覺得沒用，跟自己查資料沒差  
- 太高 → 用戶不信任，怕 Agent 亂動資金  
- **剛好** → 省力又有安全感

**實務建議：同一產品可以混用不同層級**

以 Crypto Wallet AI 助理為例：

- 查資產配置 → Level 1，自動執行  
- 提出再平衡建議 → Level 2，用戶確認後執行  
- 小額自動再平衡（設定上限）→ Level 3，有限自主  
- 大額操作 → 永遠 Level 2，強制確認
