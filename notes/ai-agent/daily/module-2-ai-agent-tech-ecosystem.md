# 模組二：AI Agent 技術生態

### Module 2: AI Agent Tech Ecosystem

學習目標：看得懂主流 Agent 技術，能跟工程師討論，不需要自己寫程式。

---

## 💡 核心心智模型：機器人比喻延伸

Agent 就像一台機器人。

- **LLM** \= 大腦晶片（提供智能與推理能力）  
- **工具 Tools** \= 手和腳（執行大腦的指令）  
- **MCP** \= 手腳與外部世界的統一連接標準（就像 USB-C）  
- **RAG** \= 機器人出門前先查筆記，補充自己不知道的知識  
- **短期記憶** \= RAM，任務中暫存，關機就消失  
- **長期記憶** \= 硬碟，跨任務都能記住  
- **語意記憶** \= 硬碟裡的「知識百科」  
- **程序記憶** \= 硬碟裡的「操作手冊與行為準則」

**關鍵洞察：**

同一顆大腦晶片（LLM），裝進不同的機器人身體（Agent 架構），就有不同的能力。 評估一個 AI 產品，不能只看晶片等級，要看整台機器人的設計。

## 2-1 主流 Agent 框架

### Major Agent Frameworks

**什麼是框架？**

框架 \= 已經幫你把 LLM、工具、記憶體的連接方式都寫好的「積木套件」。 工程師只需要組裝，不需要從零造輪子。

### 四大主流框架比較

| 框架 | 強項 | 最適合的場景 | Web3 應用 |
| :---- | :---- | :---- | :---- |
| **LangChain** | 生態最成熟、工具最多 | 需要大量工具整合 | 最常用的 Web3 Agent 基礎 |
| **LlamaIndex** | 文件處理、知識查詢 | 讓 Agent 讀懂大量資料 | 白皮書分析、鏈上數據查詢 |
| **AutoGen** | 多 Agent 互相討論 | 需要審查和辯論的任務 | 交易策略的風險審查 |
| **CrewAI** | 直覺的角色分工 | 任務分工明確的團隊協作 | DeFi 研究 \+ 分析 \+ 執行三合一 |

### 框架可以混搭

例：「自動閱讀 DeFi 白皮書 \+ 評估投資風險」

- LlamaIndex 負責讀懂文件、整理重點  
- AutoGen 負責多角度審查風險、互相討論

實際的 Web3 Agent 專案中，混搭框架很常見。

### 對 PM／BD／DevRel 的意義

- **PM：** 跟工程師討論「我們要用哪個框架」時，理解每個選擇的取捨  
- **BD：** 跟合作方說明產品基於什麼框架，知道代表什麼能力與限制  
- **DevRel：** 幫助開發者選擇適合他們 Web3 專案的框架

## 2-2 MCP（Model Context Protocol）

### What is MCP?

**沒有 MCP 之前的問題：**

每接一個新工具，工程師就要重新寫一套連接程式碼。 100 個工具 \= 寫 100 次不同的連接方式。 就像以前每支手機都要配一條專屬充電線。

**MCP 的解法：**

定義一個統一的「插頭規格」，讓所有工具都用同一種方式跟 Agent 溝通。 就像 USB-C 統一了充電接頭。

Agent

  ↓

MCP（統一標準）

  ↓

├── Google Drive

├── GitHub

├── Slack

├── 鏈上數據 API

└── 任何工具...

### MCP 的三個角色

| 角色 | 英文 | 說明 |
| :---- | :---- | :---- |
| 客戶端 | MCP Client | 發出請求的 Agent（例如 Claude） |
| 伺服器 | MCP Server | 提供工具的服務（例如 Google Drive） |
| 協議 | Protocol | 雙方溝通的統一語言規則 |

### Agent 對接 MCP 的方式

MCP 不是獨立運作的——它是建在**工具呼叫（Tool Use）** 之上的標準。

LLM 決定呼叫工具（Tool Use）

    ↓

透過 MCP 標準格式傳送請求

    ↓

外部工具收到並回傳結果

比喻：Tool Use 是你決定「我要打電話」，MCP 是電話網路的通訊標準，讓你的手機能跟對方的手機講話。

### Web3 的潛力

| Web3 工具 | 透過 MCP 可以做的事 |
| :---- | :---- |
| 鏈上數據 API | 即時查詢任何錢包、合約的狀態 |
| DEX 合約 | 直接執行 swap、存款、提款 |
| 跨鏈橋 | 自動處理跨鏈資產移動 |
| 風險評分服務 | 即時檢查合約安全性 |

⚠️ Web3 的 MCP Server 還在早期發展階段——對 BD 和 DevRel 來說是值得關注的空白市場。

## 2-3 什麼是 RAG？

### RAG: Retrieval-Augmented Generation（檢索增強生成）

**LLM 的根本限制：**

LLM 只知道訓練時學過的東西，不知道訓練之後發生的事。

**RAG 的解法：先查資料，再回答問題。**

### 三個字拆解

| 英文 | 中文 | 意思 |
| :---- | :---- | :---- |
| Retrieval | 檢索 | 去外部知識庫把相關資料找回來 |
| Augmented | 增強 | 把找回來的資料加進 LLM 的 Context |
| Generation | 生成 | LLM 根據這些資料生成回答 |

### RAG 的運作流程

用戶問問題

    ↓

① 檢索：去知識庫搜尋相關段落

    ↓

② 增強：把最相關的段落塞進 Context Window

    ↓

③ 生成：LLM 根據資料回答問題

### 沒有 RAG vs 有 RAG

|  | 沒有 RAG | 有 RAG |
| :---- | :---- | :---- |
| 知識來源 | 只有訓練時學過的 | 訓練資料 ＋ 即時外部資料 |
| 即時資訊 | ❌ 不知道 | ✅ 可以查到 |
| 私有資料 | ❌ 不知道 | ✅ 可以查公司內部文件 |
| 回答準確度 | 可能過時 | 基於最新資料 |

### RAG 知識庫是誰建的？

「外部」是對 LLM 來說的——不在 LLM 的訓練資料裡。 知識庫通常是**公司自己建造與維護**的。 重點不是誰建的，而是 LLM 原本不知道這些內容，需要靠檢索才能取得。

### RAG vs 網路搜尋

|  | RAG 知識庫 | 網路搜尋 |
| :---- | :---- | :---- |
| 資料來源 | 公司自己整理的 | 公開網路 |
| 資料品質 | 高，經過篩選 | 參差不齊 |
| 即時性 | 需要手動更新 | 即時 |
| 隱私 | 可存放私有資料 | ❌ 不能放私密資訊 |

實務上，好的 Agent 通常兩個都有：私有問題 → 查 RAG；即時公開資訊 → 網路搜尋。

### 誰決定 Agent 能不能上網搜尋？

**由「部署 Agent 的人」決定，分三個層次：**

| 層次 | 誰 | 決定什麼 |
| :---- | :---- | :---- |
| AI 公司 | Anthropic、OpenAI | LLM 本身有沒有支援工具呼叫的能力 |
| 工程師／產品團隊 | 你的公司 | 要給這個 Agent 開通哪些工具 |
| 使用者 | 你 | 在工程師允許的範圍內，決定要不要用某個工具 |

決定「這個 Agent 可以用哪些工具」，本質上就是 **PM 的產品決策**。

### RAG 實作的五個步驟（跟工程師討論用）

| 步驟 | 技術名稱 | PM 需要關心的問題 |
| :---- | :---- | :---- |
| 1\. 準備資料 | Data Preparation | 知識庫放什麼？誰負責維護更新？ |
| 2\. 切塊 | Chunking | 切太小會失去上下文，切太大會超出 Context |
| 3\. 向量化 | Embedding | 用什麼模型？支援繁體中文嗎？ |
| 4\. 向量資料庫 | Vector Database | 用哪個工具？更新頻率多久？ |
| 5\. 檢索＋生成 | Retrieval \+ Generation | 找幾段資料？準確率夠嗎？ |

### 跟工程師開會最實用的三個問題

1. 「我們的知識庫多久更新一次？用戶問到最新公告會準嗎？」  
2. 「檢索的準確率怎麼測試？有沒有 Evaluation 機制？」  
3. 「如果找不到相關資料，Agent 會怎麼處理？會亂答還是說不知道？」

### Web3 場景最重要的 RAG 應用

| 場景 | RAG 知識庫的內容 |
| :---- | :---- |
| DeFi 協議助理 | 各協議的白皮書、文件、風險參數 |
| Crypto Exchange | 最新的上幣公告、交易規則、費率表 |
| 鏈上風險偵測 | 已知詐騙合約地址資料庫、Rug Pull 案例 |
| Crypto Wallet | 用戶的歷史交易紀錄、個人偏好設定 |

### LlamaIndex 的角色

LlamaIndex 最擅長的，就是幫 Agent 建立和查詢 RAG 知識庫。 這也是為什麼「讀懂 DeFi 白皮書」的任務選 LlamaIndex。

## 2-4 LLM 與 Agent 的關係

### LLM as the Brain, Agent as the Body

### 最精準的定義

|  | LLM | Agent |
| :---- | :---- | :---- |
| 是什麼 | 語言模型，一個 AI 程式 | 一個系統架構 |
| 能做什麼 | 接收文字輸入，產生文字輸出 | 感知、決策、行動、記憶 |
| 會主動做事嗎 | ❌ 不會，等人問才回答 | ✅ 會，自己拆解任務、執行 |
| 單獨存在嗎 | ✅ 可以 | ❌ 裡面一定要有 LLM |

**一句話：LLM 是零件，Agent 是用這個零件組裝出來的機器。**

### 機器人比喻

LLM \= 大腦晶片，本身很厲害，但單獨放在桌上什麼都做不了。 Agent \= 整台機器人，把晶片裝進去，再裝上手、腳、眼睛、記憶體，才能真正行動。

同一顆大腦晶片（LLM）

    ├── 裝進客服機器人（配客服工具 \+ 客服知識庫）

    ├── 裝進交易機器人（配交易工具 \+ 市場數據）

    └── 裝進風控機器人（配風控工具 \+ 黑名單資料庫）

### 對 PM 的實際意義

評估競爭對手的 AI 產品時，要分開問兩個問題：

| 問題 | 在問什麼 |
| :---- | :---- |
| 「你們用什麼模型？」 | LLM 層：大腦晶片是誰？ |
| 「你們的 Agent 架構怎麼設計？」 | Agent 層：整台機器人怎麼組裝？ |

競爭對手說「我們升級到最新模型」，不代表產品一定比你強。 產品能力取決於：工具設計、RAG 品質、Agent 架構——三者缺一不可。

## 2-5 Agent 的記憶類型

### Types of Agent Memory

### 四種記憶完整對照

| 類型 | 英文 | 存什麼 | 機器人比喻 | Web3 例子 |
| :---- | :---- | :---- | :---- | :---- |
| 短期記憶 | Short-term Memory | 當前任務的上下文 | RAM，關機就消失 | 這次對話說的交易需求 |
| 長期記憶 | Long-term Memory | 跨任務的用戶資料 | 硬碟，關機後還在 | 用戶的風險偏好設定 |
| 語意記憶 | Semantic Memory | 事實與知識 | 硬碟裡的「知識百科」 | DeFi 協議規則、幣種資訊 |
| 程序記憶 | Procedural Memory | 行為規則與流程 | 硬碟裡的「操作手冊」 | 大額交易必須二次確認 |

### 人類記憶 vs Agent 記憶

| 類型 | 人類 | Agent |
| :---- | :---- | :---- |
| 短期記憶 | 暫時記在腦中，幾分鐘後可能忘了 | Context Window，對話結束就清空 |
| 長期記憶 | 記得朋友的名字、怎麼騎腳踏車 | 長期記憶資料庫，跨對話都能存取 |

AI 的記憶設計，本來就是參考人類認知科學來設計的。

### 程序記憶在 Web3 特別重要

程序記憶定義了 Agent 的**安全護欄**——它會做什麼、不能做什麼。

程序記憶範例（DeFi Agent）：

✅ 小於 100 USDT 的交易可以自動執行

✅ 大於 100 USDT 一定要請用戶確認

❌ 永遠不能在沒有授權的情況下轉移資產

❌ 遇到不認識的合約地址，先警告再詢問

對 PM 來說，**定義程序記憶的邊界，就是定義產品的安全護欄。**

### DeFi 投資 Agent 的記憶設計（課堂練習）

| 記憶類型 | DeFi 投資 Agent 需要的內容 |
| :---- | :---- |
| 短期記憶 | 這次任務的投資金額 |
| 長期記憶 | 用戶想買賣的幣別偏好 |
| 語意記憶 | DeFi 協議知識、各池子的風險與報酬規則 |
| 程序記憶 | 操作規則與禁止行為，例如「不可在未授權情況下動用資金」 |
